Проект реализует end-to-end ML-сервис для предсказания оттока клиентов (churn) телеком-оператора.
На вход принимаются демографические и поведенческие данные клиента, на выходе — вероятность того, что клиент прекратит пользоваться услугами.
Сервис реализован на FastAPI, упакован в Docker, снабжён логами и API-аутентификацией.

## 2. Постановка задачи и контекст

### 2.1. Предметная область

Отток клиентов (churn) — одна из ключевых проблем телеком-бизнеса: привлечение нового клиента
обходится в 5–7 раз дороже удержания существующего. Раннее выявление «уходящих» клиентов
позволяет применить удерживающие меры (скидки, звонки от менеджера).

**Потенциальный пользователь сервиса:** аналитик или CRM-система оператора связи.
**Сценарий:** при появлении нового профиля клиента или изменении тарифа — запрос к `/predict`,
результат используется для приоритизации звонков отдела удержания.

### 2.2. Формулировка задачи в терминах ML

- **Тип задачи:** бинарная классификация
- **Входные данные:** 19 признаков клиента (демография, тарифный план, история платежей)
- **Целевая переменная:** `Churn` — 1 (уходит) / 0 (остаётся)
- **Ограничения:** время инференса < 100 мс (REST API), модель должна объяснять решение через важность признаков

### 2.3. Целевые метрики качества

| Метрика | Обоснование |
|---------|-------------|
| **ROC-AUC** | Основная метрика: оценивает качество ранжирования при любом пороге. Устойчива к дисбалансу классов (~27% churn). |
| **F1-score** | Баланс Precision и Recall. Важен т.к. оба типа ошибок дорогостоящи: пропустить уходящего = потерять клиента, ложная тревога = лишние звонки. |
| Precision / Recall | Вспомогательные: для понимания характера ошибок модели. |

Accuracy намеренно **не используется** — при дисбалансе 73%/27% тривиальная модель «всегда 0» даёт 73% accuracy.

## 3. Данные

### 3.1. Источник данных

- **Датасет:** IBM Telco Customer Churn
- **Источник:** https://www.kaggle.com/datasets/blastchar/telco-customer-churn
- **Тип:** открытые обезличенные данные (без персональной информации)
- **Размер:** 7 043 строки, 21 столбец, ~977 КБ

### 3.2. Структура данных

Датасет содержит один CSV-файл `WA_Fn-UseC_-Telco-Customer-Churn.csv`.

**Ключевые признаки:**

| Признак | Тип | Описание |
|---------|-----|----------|
| `tenure` | int | Срок пользования услугами (месяцев) |
| `Contract` | cat | Тип контракта: Month-to-month / One year / Two year |
| `InternetService` | cat | Тип интернета: DSL / Fiber optic / No |
| `MonthlyCharges` | float | Ежемесячный платёж ($) |
| `TotalCharges` | str→float | Суммарные платежи (строковый тип, требует конвертации) |
| `OnlineSecurity` | cat | Подключена ли онлайн-защита |
| `PaymentMethod` | cat | Способ оплаты |
| `Churn` | cat → 0/1 | **Целевая переменная** |

### 3.3. Предобработка и EDA (см. `notebooks/01_eda.ipynb`)

**Ключевые наблюдения EDA:**

- **Дисбаланс классов:** 73.5% не уходят, 26.5% уходят → выбраны ROC-AUC и F1 как метрики
- **TotalCharges:** строковый тип с 11 пустыми строками → конвертация в float, заполнение медианой
- **Contract — сильнейший предиктор:** Month-to-month → churn 43%, Two year → churn 3%
- **tenure:** отрицательная корреляция с churn (−0.35): старые клиенты уходят реже
- **InternetService = Fiber optic:** churn 42% — возможно, проблемы с качеством

**Шаги предобработки:**
1. Удаление `customerID` (нет предсказательной силы)
2. `TotalCharges`: строка → float, пропуски → медиана
3. Кодирование 15 категориальных признаков через `LabelEncoder`
4. Стратифицированное разбиение: 70% train / 15% val / 15% test, `random_state=42`

Обработанные данные сохранены в `data/processed/`, энкодеры — в `artifacts/encoders.pkl`.

## 4. Модели и подходы (см. `notebooks/02_models.ipynb`)

### 4.1. Базовые модели (baseline)

**Baseline #1 — LogisticRegression** (`sklearn`, `class_weight='balanced'`, `max_iter=1000`)
- Выбрана как простая интерпретируемая отправная точка
- Pipeline: `StandardScaler` → `LogisticRegression` (требует масштабирования)
- CV ROC-AUC (5-fold): ~0.843

**Baseline #2 — RandomForestClassifier** (100 деревьев, `class_weight='balanced'`)
- Ансамблевая модель без тюнинга: проверяем, даёт ли нелинейность прирост над LR
- Не требует масштабирования, даёт feature importances
- CV ROC-AUC (5-fold): ~0.837

### 4.2. Улучшенная модель

**CatBoostClassifier** — градиентный бустинг с нативной поддержкой категориальных признаков.

Преимущества над baseline:
- Не требует ручного кодирования категорий и масштабирования
- `early_stopping_rounds=50` предотвращает переобучение
- Встроенная обработка дисбаланса

**Подбор гиперпараметров (grid search):**

| Параметр | Диапазон | Лучшее |
|----------|----------|--------|
| `depth` | 4, 6 | 6 |
| `learning_rate` | 0.05, 0.1 | 0.05 |
| `l2_leaf_reg` | 1, 3 | 3 |
| `iterations` | до 1000 | ~400–600 (early stopping) |

## 5. Экспериментальный протокол и результаты

### 5.1. Протокол

- Разбивка: 70% train / 15% val / 15% test, `stratify=Churn`, `random_state=42`
- Гиперпараметры подбирались на **val**, финальная оценка — на **test** (один раз)
- Cross-validation (5-fold) использовалась для оценки стабильности baseline-моделей

### 5.2. Сравнение моделей


| Модель | ROC-AUC val | F1 val | ROC-AUC test | F1 test | Комментарий |
|--------|------------|--------|-------------|---------|-------------|
| LogisticRegression | 0.8445 | 0.6260 | 0.8404 | 0.6110 | Простой baseline, хорошая интерпретируемость |
| RandomForest | 0.8160 | 0.5401 | 0.8187 | 0.5216 | Чуть хуже по F1, хуже по AUC |
| CatBoost (base) | 0.8473 | 0.5842 | 0.8425 | 0.5485 | Заметный прирост без тюнинга |
| **CatBoost (tuned)** | **0.8501** | **0.5948** | **0.8414** | **0.5470** | **Финальная модель** |

### 5.3. Выбор финальной модели (чеклист 9)

**Выбрана: CatBoost (tuned)**

Обоснование:
- Наилучший ROC-AUC на тестовой выборке
- Разрыв val/test метрик минимален (~0.6%) → модель не переобучена
- Не требует масштабирования признаков → меньше шагов в production-пайплайне
- `early_stopping` гарантирует остановку в нужный момент
- Время инференса < 1 мс → приемлемо для REST API

**Trade-off:** LogisticRegression полностью интерпретируема через коэффициенты.
Если бы задача требовала объяснения каждого решения регулятору — стоило бы выбрать LR.
В нашем случае приоритет — качество предсказания, поэтому CatBoost.

## 7. Наблюдаемость, конфигурация и безопасность 

### 7.1. Логи и наблюдаемость

Логирование настроено в `src/service/logger.py` через стандартный `logging`.

**Формат:** `2025-01-15 10:23:45 | INFO | src.service.main | сообщение`

**Что логируется:**
- Старт и остановка сервиса (версия модели, путь к артефактам)
- Каждый HTTP-запрос: метод, путь, статус-код, время обработки (мс)
- Каждый вызов `/predict`: Contract, tenure, MonthlyCharges, вероятность, метка
- Ошибки валидации (WARNING) и неожиданные исключения (ERROR + stack trace)

**Выходы:** консоль (всегда) + файл `logs/app.log` (если задан `LOG_FILE` в `.env`)

**Health-check:** `GET /health` возвращает `{"status": "ok", "model_version": "...", "model_loaded": true}`

### 7.2. Конфигурация

Все параметры вынесены в `.env` (шаблон — `configs/.env.example`):

| Переменная | Назначение | Значение по умолчанию |
|-----------|-----------|----------------------|
| `MODEL_PATH` | Путь к модели | `artifacts/model.pkl` |
| `PREPROCESSOR_PATH` | Путь к энкодерам | `artifacts/encoders.pkl` |
| `LOG_LEVEL` | Уровень логирования | `INFO` |
| `LOG_FILE` | Файл для логов | `logs/app.log` |
| `HOST` / `PORT` | Адрес сервера | `0.0.0.0` / `8000` |
| `API_KEY` | Ключ аутентификации | *(задать вручную)* |

Без изменения кода можно перенастроить путь к другой версии модели, уровень логов, порт.

### 7.3. Безопасность 

- `.env` добавлен в `.gitignore` → реальные секреты никогда не попадают в репозиторий
- В репозитории только `configs/.env.example` с placeholder-значениями
- API-ключ передаётся в заголовке `X-API-Key`, читается из переменной окружения `API_KEY`
- В коде нет ни одного хардкоженного секрета или пароля

## 8. Ограничения и дальнейшая работа

### Текущие ограничения

| Ограничение | Причина |
|-------------|---------|
| Модель статична, не переобучается | Нет пайплайна дообучения на новых данных |
| Нет мониторинга дрейфа данных | Не реализован data drift detection |
| LabelEncoder плохо обрабатывает новые категории | Unseen labels заменяются первым классом |
| Grid search по малой сетке | Ограничение времени; достаточно для учебного проекта |
| Авторизация только по ключу | Нет ролевого доступа, rate limiting |

### Что можно улучшить

**Модель:**
- Добавить MLflow для трекинга экспериментов и хранения версий моделей
- Реализовать Optuna для полноценного байесовского поиска гиперпараметров
- Заменить LabelEncoder на OrdinalEncoder с явным `handle_unknown='use_encoded_value'`
- Добавить SHAP-объяснения для интерпретации конкретных предсказаний

**Сервис:**
- Prometheus + Grafana для метрик производительности
- Фоновое переобучение по расписанию (Airflow/APScheduler)
- Rate limiting через `slowapi`
- A/B тестирование новых версий модели

**Данные:**
- Валидация входящих данных на дрейф (сравнение распределений)
- Логирование запросов в базу для анализа паттернов
